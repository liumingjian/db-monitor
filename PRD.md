# PRD: MySQL-First Internal DB Monitoring Platform

## Problem Statement

当前可参考的 Lepus v3.8 只能提供领域知识，不能作为新产品的运行时基础。旧系统的核心问题不是 PHP 或 Python 本身，而是 UI、配置、采集、聚合、告警全部通过同一个 MySQL 控制库强耦合，导致边界不清、演进困难、运行脆弱、模式漂移严重。

Boss 已明确确认，新产品没有历史迁移负担，也不需要兼容旧运行时。因此，本产品需要以绿地方式重建一个内部、单租户、MySQL-first 的数据库监控平台，优先解决实例接入、核心指标采集、统一展示、基础告警和可维护架构问题，而不是复制 Lepus 的全引擎和全功能包袱。

## Solution

构建一个单体控制面加独立数据面的新监控平台。控制面以 FastAPI 作为唯一业务 API 边界，承载认证、RBAC、实例管理、规则管理、告警历史、系统设置和仪表盘聚合能力；前端使用 Next.js 作为纯 UI 层，通过 OpenAPI 生成的类型化客户端访问后端；数据面采用 scheduler、typed workers、rule-engine、notifier 的显式流水线，将采集、规则判断和通知发送彻底拆开。

存储从第一天开始分层：PostgreSQL 负责控制数据，ClickHouse 负责时序和分析数据，Redis 只负责任务分发。第一阶段仅交付内部可用的 MySQL-first 版本，采用 agentless 方式采集高价值核心指标，支持基础规则告警和常用趋势查询，形成一个可扩展但不过度设计的起点。

## User Stories

1. 作为内部管理员，我希望使用账号密码登录系统，这样我可以安全进入监控平台。
2. 作为内部管理员，我希望系统保存会话状态，这样我在日常使用中不必频繁重新认证。
3. 作为平台管理员，我希望能够管理用户、角色和权限，这样我可以控制谁能查看、编辑和管理监控资源。
4. 作为只读用户，我希望只能查看被授权的实例和仪表盘，这样我不会误改配置。
5. 作为 DBA，我希望新增一个 MySQL 实例并填写连接配置，这样系统可以开始采集该实例的监控数据。
6. 作为 DBA，我希望在实例接入时立即验证连接是否可用，这样我能快速发现配置错误。
7. 作为 DBA，我希望查看实例列表及其当前状态，这样我可以快速识别异常实例。
8. 作为 DBA，我希望按名称、环境、标签或状态筛选实例，这样我可以快速定位目标对象。
9. 作为 DBA，我希望查看实例详情页，这样我可以集中查看该实例的基础信息、核心指标和告警记录。
10. 作为 DBA，我希望看到实例的可用性、角色、版本和 uptime，这样我可以快速判断基础健康状态。
11. 作为 DBA，我希望查看 threads connected 和 threads running 趋势，这样我可以识别连接压力和并发问题。
12. 作为 DBA，我希望查看 QPS、TPS 和网络流量趋势，这样我可以判断业务负载变化。
13. 作为 DBA，我希望查看核心 InnoDB 指标趋势，这样我可以判断存储引擎健康情况。
14. 作为 DBA，我希望查看复制状态和复制延迟，这样我可以发现主从复制异常。
15. 作为 DBA，我希望查看基础利用率类指标，这样我可以评估实例容量风险。
16. 作为 DBA，我希望系统对指标做统一建模，同时保留 MySQL 扩展字段，这样核心能力一致且不过度抽象。
17. 作为告警配置人员，我希望为 MySQL 实例配置基础阈值规则，这样系统能在明显异常时及时告警。
18. 作为告警配置人员，我希望定义规则的启停状态和作用范围，这样我可以控制哪些实例受哪些规则约束。
19. 作为值班人员，我希望当规则命中后产生清晰的告警事件，这样我能快速进入处理流程。
20. 作为值班人员，我希望看到告警列表，并按状态、级别、时间和实例筛选，这样我可以高效排查当班问题。
21. 作为值班人员，我希望查看单条告警详情，这样我可以知道触发条件、关联实例、触发时间和恢复情况。
22. 作为平台管理员，我希望采集、规则计算和通知发送是分离的阶段，这样我可以独立排查每个环节的问题。
23. 作为平台管理员，我希望采集器只负责采集和写入结果，而不是直接发通知，这样系统边界清晰且更易调试。
24. 作为平台管理员，我希望任务调度和 worker 执行有明确职责分工，这样运行时行为稳定且容易扩容。
25. 作为平台管理员，我希望 Redis 仅承担任务分发职责，这样基础设施责任单一、问题更容易定位。
26. 作为产品负责人，我希望前端只负责展示和交互，这样业务规则不会散落在 UI 层。
27. 作为后端开发者，我希望 API 内部遵循 router -> service -> repository 分层，这样我可以在不牵动全局的情况下演进业务逻辑。
28. 作为前端开发者，我希望通过 OpenAPI 生成类型化客户端，这样前后端契约可以稳定协作并减少手写错误。
29. 作为前端开发者，我希望列表、详情、告警和趋势页面都通过统一查询层取数，这样状态管理和缓存策略更一致。
30. 作为系统管理员，我希望通过系统设置管理全局行为和通知参数，这样运维配置集中且可审计。
31. 作为审计人员，我希望关键操作被记录为审计日志，这样我可以追踪谁在什么时间修改了什么配置。
32. 作为内部团队，我希望该系统首先只服务单租户内部场景，这样 phase one 可以专注核心价值而不是多租户复杂度。
33. 作为开发者，我希望项目采用 monorepo 组织结构，这样共享类型、UI 组件和 API 客户端可以统一管理。
34. 作为开发者，我希望本地开发优先，不强依赖容器化运行，这样日常开发和调试成本更低。
35. 作为测试工程师，我希望核心 API、服务层、仓储层和数据访问路径从第一天就可测试，这样质量门禁不是后补。
36. 作为团队负责人，我希望通过 lint、type check、契约检查和最小端到端冒烟测试来拦截回归，这样第一阶段也具备工程可控性。

## Implementation Decisions

- 产品以绿地方式重建，不复用 Lepus 旧运行时，也不承担向后兼容义务。
- Phase one 聚焦内部单租户的 MySQL-first 版本，不做多引擎并行推进。
- 控制面采用模块化单体，FastAPI 是唯一业务 API 边界。
- 前端使用 Next.js、React、Tailwind CSS v4、shadcn/ui、ECharts 和 TanStack Query，但前端不持有业务规则或数据库访问逻辑。
- API 内部统一采用 router -> service -> repository 分层，避免控制器直接承载复杂业务和 SQL。
- API 契约采用 OpenAPI-first，并生成类型化客户端供前端使用，避免手工维护接口类型。
- 控制数据存储使用 PostgreSQL，承载用户、RBAC、资产、规则、系统设置、审计和告警记录。
- 分析与时序数据存储使用 ClickHouse，承载指标快照、历史趋势和聚合查询。
- Redis 只承担任务分发，不承担业务真相存储或隐式控制逻辑。
- 运行时拆分为 scheduler、mysql worker、rule engine、notifier 四类进程，职责清晰且可独立扩展。
- MySQL phase one 采用 agentless 采集，避免第一阶段引入客户端代理生命周期管理。
- 指标模型采用统一核心字段加引擎扩展字段的模式，既统一查询入口，又保留 MySQL 领域表达力。
- 仪表盘和实例详情只覆盖高价值核心指标，不在第一阶段承载慢查询深挖、备份监控或复杂报表。
- 告警链路必须显式拆分为 collection -> rule evaluation -> notification，采集器不得直接发送通知。
- 规则系统第一阶段仅支持基础规则集，不引入复杂 DSL 或过度灵活但难维护的规则语义。
- 身份认证和 RBAC 需要重建，不延用 Lepus 的旧权限模型。
- 推荐优先沉淀以下深模块：认证与会话模块、RBAC 授权模块、MySQL 资产接入模块、指标归一化写入模块、仪表盘聚合查询模块、规则评估模块、通知分发模块。
- 审计日志应覆盖登录、实例变更、规则变更、系统设置变更等关键控制面操作。
- Monorepo 需要清晰区分应用层与共享包，避免未来再次形成跨边界耦合。
- 新系统只复用 Lepus 的领域词汇、指标启发和历史问题教训，不复用其表结构、进程模型或控制流。

## Testing Decisions

- 好的测试应验证外部行为和业务结果，而不是绑定内部实现细节、调用顺序或私有结构。
- 认证与会话模块需要覆盖登录成功、登录失败、会话失效和权限拒绝等外部行为。
- RBAC 授权模块需要覆盖角色授权、资源访问控制和最小权限边界。
- MySQL 资产接入模块需要覆盖实例创建、连接校验、配置修改和状态查询。
- 服务层与仓储层需要覆盖规则管理、告警查询、系统设置和审计日志写入。
- 指标写入与查询链路需要通过数据访问集成测试验证 PostgreSQL、ClickHouse 和 Redis 的边界协作。
- 规则评估模块需要覆盖触发、抑制、恢复和重复事件处理等可观察结果。
- 通知模块需要覆盖事件到通知请求的转换和失败暴露，不允许静默降级伪成功。
- OpenAPI 需要持续生成并执行契约差异检查，确保前后端接口变更可见。
- 前端至少需要保留最小端到端冒烟测试，覆盖登录、实例接入、仪表盘查看、规则创建和告警查看的主路径。
- 前端必须执行类型检查和 lint，后端必须执行 lint、类型检查和单元测试，形成 phase one 的最低质量门禁。
- 当前仓库中可参考的“类似测试”主要是 Lepus 里的连接性与驱动脚本，这些只可作为领域参考，不应成为新系统测试标准；新项目应建立以 service、repository、contract 和 smoke test 为核心的测试体系。

## Out of Scope

- Oracle、Redis、MongoDB、SQL Server、OS 等多引擎能力
- 历史 Lepus 数据或配置迁移
- 向后兼容 Lepus 运行时、脚本和表结构
- 慢查询深度分析
- Bigtable 扫描
- 备份监控
- Binlog 维护
- 高级报表生成器
- 复杂规则 DSL
- 多租户架构
- 容器优先的部署体系
- 前端直接访问数据库或承载业务逻辑

## Further Notes

- 本 PRD 以 `RESEARCH.md` 中已确认结论为准，目标是给 phase one 提供稳定边界，而不是一次性覆盖未来所有监控能力。
- 本阶段最重要的成功标准不是“功能像 Lepus 一样多”，而是“边界清晰、核心链路可运行、后续扩展不再受共享控制库耦合拖累”。
- 如后续进入分解实施，应继续围绕深模块拆分任务，优先保证认证/RBAC、资产接入、指标管道、规则评估和告警查询这些核心模块可独立测试。
